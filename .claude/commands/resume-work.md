# 作業の再開（チェックポイント復元）

**重要:** このコマンドは、`/add-feature` で開始した作業が中断した場合に再開するためのコマンドです。ユーザーの介入なしに、チェックポイントを読んで未完了箇所から自動実行します。

**引数:** ステアリングディレクトリ名（例: `/resume-work 20250227-add-user-profile`）
引数を省略した場合は、`.steering/` 内の最新ディレクトリを自動選択します。

---

## ステップ0: Git ログによる状況把握（提案F）

作業再開前に、Git ログで「何が起きたか」を確認する。Git ログは最高信頼度の記録。

```bash
Bash('git log --oneline -20')
```

確認事項:
1. 前回セッションで何のコミットが行われたか
2. `.steering/` ディレクトリへの変更がコミットされているか
3. 未コミットの変更がないか（`git status`）

ユーザーに確認結果を報告する:
```
「Git ログを確認しました。
直近のコミット: [コミットメッセージ]
未コミットの変更: [あり/なし]
```

---

## ステップ1: チェックポイント読み込み

1. 引数が指定された場合: `.steering/[引数のディレクトリ名]/` を対象とする。
2. 引数が省略された場合: 最新のステアリングディレクトリを自動選択する:
  ```bash
  Bash('ls -t .steering/ | grep -v .gitkeep | head -1')
  ```
3. 以下のファイルを読み込む:
  - `.steering/[対象ディレクトリ]/tasklist.md`
  - `.steering/[対象ディレクトリ]/requirements.md`
  - `.steering/[対象ディレクトリ]/design.md`

## ステップ2: 再開ポイントの特定

tasklist.md を分析し、以下を特定する:

1. **完了済みフェーズ**: CP（チェックポイント）の「完了日時」が記録済みのフェーズ
2. **再開フェーズ**: 未完了タスク（`[ ]`）が最初に現れるフェーズ
3. **再開タスク**: そのフェーズ内で最初の未完了タスク

**判定ルール**:
- CP の「完了日時」フィールドが `{YYYY-MM-DD HH:MM}` のままのもの → 未完了
- 「完了日時」に実際の日時が記録されているもの → 完了済み
- `[ ]` が残っているタスク → 再開ポイント
- 品質ゲート（フェーズ3）が未完了 → ステップ5（品質ゲート）から再開

ユーザーに再開ポイントを報告する:
```
「[ディレクトリ名] の作業を再開します。
再開ポイント: [フェーズ名] > [タスク名]
完了済みフェーズ: [フェーズリスト]
未完了タスク数: [N]件
```

## ステップ3: コンテキスト復元

1. `CLAUDE.md` を読む。
2. `docs/` 内の関連ドキュメントを確認する。
3. `design.md` の実装アプローチを再確認する。
4. 再開フェーズの直前の CP Evidence を確認し、どこまで実装されているかを把握する。

## ステップ4: 実装の再開

再開ポイントから `/add-feature` と同じ実装ループを実行する。

**ループ開始:**

1. `tasklist.md` の再開ポイント以降の未完了タスクを特定する。
2. 未完了タスクが存在しない場合:
  - 全タスクが完了しているが品質ゲート（フェーズ3）が未実施 → ステップ5（品質ゲート）へ
  - 全タスクと品質ゲートが完了している場合 → ステップ6（振り返り）へ
3. 未完了タスクを 1 つ実行する。`Skill('steering')` を**実装モード**で使用する。
4. 完了後、`Edit` ツールで `tasklist.md` を更新する（`[ ]` → `[x]`）。
5. **ステップ4の先頭に戻り繰り返す。**

---
### ※ 例外処理ルール ※

`/add-feature` と同じ例外処理ルールを適用する:
- **ルールA**: タスクが大きすぎる場合 → サブタスクに分割してループ継続
- **ルールB**: 技術的理由でタスクが不要 → `[x] ~~タスク名~~（理由: ...）` の形式でスキップ
- **❌ 絶対禁止**: 理由なしのスキップ・ユーザーへの判断委譲

---

## ステップ5: 品質ゲート検証

`/add-feature` の **ステップ7** と全く同じ手順で品質ゲートを実行する:

1. `docs/guidelines/definition-of-done.md` を読み、適用するゲートと合格条件を確認する。
2. 反復カウンターを 0 に初期化する（最大 3 回）。
3. ゲートの実行要否を確認する（`playwright.config.ts`, `tests/vrt/__screenshots__/`, `tests/a11y/`, `.claude/agents/ui-ux-validator.md` の存在確認）。
4. Gate 1〜5 を設定済みの場合のみ実行する。
5. 失敗時は修正→再検証ループ（最大 3 回）。
6. 3 回失敗した場合は `.steering/[タスク名]/failure-report.md` を生成してユーザーにエスカレーション。

## ステップ6: 振り返りとドキュメント更新

`/add-feature` の **ステップ8** と全く同じ手順で振り返りを実施する:

1. `Skill('steering')` を**振り返りモード**で実行し、`tasklist.md` に申し送り事項を記載する。
2. 今回の変更が永続ドキュメント（`docs/`）に影響するか判断し、必要なら更新する。

## 完了条件

`/add-feature` と同じ完了条件を全て満たすこと:
- `tasklist.md` の全タスクが完了状態
- `docs/guidelines/definition-of-done.md` の全適用ゲートがパスしている
- `tasklist.md` に申し送り事項が記載されている

## 注意事項

- **ステアリングファイルは新規作成しない**（既存の requirements.md、design.md は変更しない）
- **前の CP より前のタスクを遡って変更しない**（前のチェックポイントは確定済みとして扱う）
- tasklist.md への Evidence 追記・タスク完了更新は実施してよい
