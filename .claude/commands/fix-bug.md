# バグ修正フロー (fix-bug)

**基本方針:** 推測で修正しない。再現手順が確立できない場合は作業を停止してユーザーに報告する。

**引数:** バグの説明 (例: `/fix-bug ログイン後にリダイレクトされない`)

---

## ステップ1: 事象の整理

1. `CLAUDE.md` を読み、プロジェクトの全体像を把握する。
2. 報告されたバグを以下の形式で整理する:
   - **いつ**: どの操作・イベント・条件で発生するか
   - **どこで**: どの画面・API・機能で発生するか
   - **何が起きるか**: 期待される動作 vs 実際の動作
   - **再現率**: 常に発生するか・条件次第か
3. `docs/glossary.md` が存在する場合は関連する用語を確認する。

---

## ステップ2: 再現手順の確立

**このステップが最も重要。再現手順がなければ修正は始められない。**

1. 以下の優先順位で再現手順を構築する:
   - 失敗するユニット / 統合 / E2E テスト
   - curl / HTTP スクリプト（API バグの場合）
   - CLI 実行 with フィクスチャ入力
   - Playwright / 自動ブラウザスクリプト
   - 最小再現用のスクリプト・ハーネス

2. **再現手順が確立できた場合**: 手順を記録してステップ3へ進む。
3. **再現手順が確立できない場合**: 作業を停止し、`AskUserQuestion` でユーザーに以下を提示する:
   - 試みた方法と結果
   - 再現に必要な情報（環境・データ・タイミング）
   - 取得が必要なアーティファクト（ログ・HAR ファイル・スタックトレース）

> ⚠️ 再現手順なしに推測で修正しないこと。

---

## ステップ3: 最小再現の作成

1. 再現手順から不要な要素を削ぎ落とし、バグが再現する最小のケースを特定する。
2. 最小再現を記録する（コマンド・テストコード・操作手順）。
3. 最小再現でバグが確実に再現することを確認する。

---

## ステップ4: 失敗するテストまたは確認手順の作成

**修正前に「失敗する状態」を確認する（red-green-refactor の red フェーズ）。**

1. 以下のいずれかを作成する:
   - **テストコードがある場合**: バグを再現する失敗テストを作成し、失敗を確認する
   - **テストが困難な場合**: `playwright-cli` または手動操作で失敗を確認し、手順を記録する
2. この「失敗の記録」が修正後の検証基準になる。

---

## ステップ5: 仮説の列挙

1. **3〜5 個の仮説**を生成する。単一仮説に固執しない。
2. 各仮説は **falsifiable（反証可能）** な形式で記述する:
   - 形式: 「もし X が原因なら、Y を変更するとバグが消える / Z を変更するとバグが悪化する」
3. `docs/adr/` が存在する場合、関連する設計判断を確認して仮説に反映する。
4. 仮説をランク付けし、最も可能性の高いものから検証する。

---

## ステップ6: ログ・計測の追加（必要な場合）

1. 仮説の検証に必要なログ・メトリクスを追加する。
2. **タグ付きデバッグログ**を使用する（例: `[DEBUG-a4f2]`）。タグがあれば後で一括削除できる。
3. 「ログを全部出力して grep」はしない。仮説の境界に絞る。

---

## ステップ7: 最小修正

1. 検証した仮説に基づき、**最小の変更**でバグを修正する。
2. 修正範囲を必要最低限に抑える（バグ修正と無関係なリファクタリングは別タスクにする）。
3. 高リスク変更（認証・DBスキーマ・外部API・ファイル削除・大量変更）が必要な場合は、`AskUserQuestion` でユーザーに確認する。
4. ステップ6 で追加したデバッグログを削除する（タグで grep して確認）。

---

## ステップ8: 回帰テストと動作確認

1. **ステップ4 で作成したテスト / 確認手順**を再実行し、バグが修正されたことを確認する。
2. `Skill('testing')` を実行し、既存機能が壊れていないかリグレッションチェックを行う。
   - `playwright-cli console error` と `playwright-cli network` でエラー 0 件を確認する
   - エラーが検出された場合は作業を中断し、修正タスクを追加してから戻る
3. 元の再現手順でバグが再現しないことを確認する。

---

## ステップ9: 記録

1. **ステアリングファイルが存在する場合**: `.steering/[タスク名]/` に修正内容を記録する。
2. **設計上の問題が根本原因の場合**: `AskUserQuestion` でユーザーに確認し、必要なら `docs/adr/` に ADR として記録する。
3. 修正内容とその根拠をコミットメッセージに記述する。

---

## Git 操作

```bash
git status
```

変更がある場合:
- センシティブファイルを除外してステージング（`.env`, `*.key`, `credentials*` を除外）
- コミットメッセージ形式: `fix: [バグの概要] - [根本原因の要約]`
- `AskUserQuestion` でユーザーに push するか確認する

---

## 禁止事項

- **再現手順なしに推測で修正しない**
- **テストまたは確認手順なしに完了扱いにしない**
- **実行できなかった検証を実行済みのように報告しない**
- **デバッグログを削除せずにコミットしない**
