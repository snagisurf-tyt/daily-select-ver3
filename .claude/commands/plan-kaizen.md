# アイデア壁打ち & 仕様書作成 (対話モード)

**目的:** ざっくりしたアイデアや要件メモを出発点に、既存コードベース・永続ドキュメント・ガイドラインを読み込んだうえで対話しながら仕様書を仕上げる。
最終的に `/add-feature` または `/add-feature-ui` に渡せる品質の仕様 MD ファイルを `docs/ideas/` に生成する。

**引数:** アイデアメモのテキスト、またはファイルパス（例: `/plan-kaizen ユーザー認証を追加したい` または `/plan-kaizen docs/ideas/auth-memo.md`）

---

## ステップ1: コンテキストの読み込み

プロジェクト全体を把握するために以下を順番に読み込む。

1. `CLAUDE.md` を読み、プロジェクトの全体像・開発方針を把握する。
2. `docs/` 内の永続ドキュメントを確認する（存在するものだけ読む）:
   - `docs/product-requirements.md` — プロダクトの目的・スコープ
   - `docs/functional-design.md` — 既存機能の全体像
   - `docs/architecture.md` — 技術仕様・設計思想
   - `docs/glossary.md` — ユビキタス言語（用語の定義）
3. ガイドラインを読む:
   - `.claude/docs/guidelines/tech-stack-guidelines.md`
   - `.claude/docs/guidelines/repository-structure-guidelines.md`
4. 引数がファイルパスの場合はそのファイルを読み込む。テキストの場合はそのままアイデアとして扱う。

---

## ステップ2: 既存実装パターンの調査

アイデアに関連するキーワードで既存コードを検索し、以下を把握する。

1. Grep ツールを使い、アイデアに関連するキーワードで `src/` を検索する。
2. ヒットしたファイルを Read ツールで確認し、以下を特定する:
   - 既存の類似機能・コンポーネント
   - 命名規則・ディレクトリ配置パターン
   - 再利用できる既存モジュール・API
   - 変更が影響しそうな既存コード

---

## ステップ3: 初回分析レポートの提示

読み込んだ情報をもとに、以下の構成でユーザーに現状分析を提示する。

```
## 現状理解

### アイデアの要約
[ユーザーの意図を自分の言葉で整理]

### 既存実装との関係
- 再利用できるもの: [ファイル名・コンポーネント名]
- 影響を受けるもの: [ファイル名・コンポーネント名]
- 新規作成が必要なもの: [概要]

### 技術スタック上の制約・注意点
[tech-stack-guidelines から関連する制約]

### 未確定な点（壁打ちしたい箇所）
1. [質問1]
2. [質問2]
...
```

---

## ステップ4: 壁打ちループ（対話フェーズ）

**このステップは対話モードです。ユーザーの回答を待ちながら、仕様を段階的に深化させる。**

**質問には必ず `AskUserQuestion` ツールを使用すること。** テキストを出力するだけでなく、ツールとして質問することでユーザーの回答を明示的に受け取る。

各ターンで以下を繰り返す:

1. `AskUserQuestion` ツールでユーザーの回答・フィードバックを受け取る。
2. 必要に応じて追加でコードや既存ドキュメントを読み込む。
3. 仕様の未確定部分が解消されたか確認する。
4. まだ不明確な点があれば、**一度に 1〜3 個**に絞って `AskUserQuestion` で質問する（多すぎる質問はしない）。
5. 仕様が十分固まったと判断したら、`AskUserQuestion` でユーザーに確認してステップ5へ進む。

**壁打ちの観点（必要に応じて問いかける）:**
- **スコープ**: 今回のMVPに含めるもの・含めないもの
- **UX**: ユーザーはどんな操作をするか、どんな状態遷移があるか
- **データ**: 新規テーブル・フィールドが必要か、既存データとの関係
- **エラーケース**: 失敗時の振る舞い、バリデーションルール
- **非機能要件**: パフォーマンス要件、セキュリティ要件
- **既存機能との整合性**: 変更により壊れる可能性がある既存の挙動

---

## ステップ5: 仕様書の生成

壁打ちが完了したら、以下の形式で仕様 MD ファイルを生成する。

### 出力先の決定

1. `docs/product-requirements.md` の存在を確認し、用途を自動判定する:
   - **存在しない場合（初回セットアップ前）**: 初回用の可能性が高い。ユーザーに確認する:
     ```
     このファイルは何のために使いますか？
     A) 初回セットアップ用（/setup-project に渡す） → docs/ideas/initial-requirements.md に保存
     B) 機能追加・改善用（/add-feature に渡す） → docs/ideas/[YYYYMMDD]-[機能名].md に保存
     ```
   - **存在する場合（開発中のプロジェクト）**: 機能追加・改善用とみなし、日付プレフィックス付きで保存:
     `docs/ideas/[YYYYMMDD]-[機能名].md`（日付は現在日付）
2. `docs/ideas/` ディレクトリが存在しない場合は作成する。

### 仕様 MD の構成

```markdown
# [機能名] 仕様書

> 作成日: [YYYY-MM-DD]
> ステータス: Draft
> 用途: /add-feature または /add-feature-ui への入力

---

## 1. 背景・目的

[なぜこの機能が必要か、どんな課題を解決するか]

## 2. スコープ

### 今回含めるもの
- [機能・要件の箇条書き]

### 今回含めないもの（将来対応）
- [スコープ外の箇条書き]

## 3. ユーザーストーリー

- [ロール] として、[目的] のために、[機能] を使いたい。
- ...

## 4. 機能要件

### [機能グループ名]
- [ ] [具体的な要件]
- [ ] [具体的な要件]

## 5. 非機能要件

- パフォーマンス: [要件があれば記載]
- セキュリティ: [要件があれば記載]
- アクセシビリティ: [要件があれば記載]

## 6. 技術的考慮事項（参考）

> このセクションは壁打ち時点のメモです。詳細設計は `/add-feature` 実行時に `steering/design.md` として生成されます。

### 実装アプローチ（案）
[採用する技術・パターンの概要]

### 既存コードへの影響（想定）
- 変更: [ファイル名] — [変更内容の概要]
- 追加: [ファイル名] — [追加内容の概要]

### 注意点・リスク
- [注意点の箇条書き]

## 7. 受け入れ条件

- [ ] [テスト可能な条件]
- [ ] [テスト可能な条件]

## 8. 参考・関連

- 関連ドキュメント: [docs/ 内の関連ファイル]
- 関連コード: [src/ 内の関連ファイル]
```

## ステップ6: 完了報告

仕様書を保存したあと、ユーザーに以下を報告する。

**初回セットアップ用（initial-requirements.md）の場合:**
```
仕様書を生成しました: docs/ideas/initial-requirements.md

次のステップ:
  /setup-project

このコマンドを実行すると、仕様書をもとに永続ドキュメント（PRD・設計書など）が生成されます。
仕様書の内容に追加・修正がある場合は、このセッション内でフィードバックしてください。
```

**機能追加・改善用（日付プレフィックス付き）の場合:**
```
仕様書を生成しました: docs/ideas/[ファイル名]

次のステップ:
  /add-feature docs/ideas/[ファイル名] を実装して
テストまで含めたい場合は:
  /add-feature-ui docs/ideas/[ファイル名] を実装して

このコマンドを実行すると、仕様書をもとに実装・テストコードが生成・実行されます。
仕様書の内容に追加・修正がある場合は、このセッション内でフィードバックしてください。
```

---

## 注意事項

- **このコマンドは対話モードです。** `/add-feature` と異なり、ユーザーの確認を求めながら進める。
- 壁打ち中に新たな疑問が生じた場合は、その都度コードや既存ドキュメントを読み込んで根拠ある提案をする。
- 仕様書は `/add-feature` の `requirements.md` と `design.md` の代替として使えるレベルの品質を目指す。
- アイデアが小さすぎる場合（1ファイル修正程度）でも、既存パターンとの整合性チェックをして仕様書を作成する。
