## 背景

本プロジェクトの仕様駆動開発（SDD）テンプレートを、Harness Engineering のベストプラクティス記事と比較し、7つの改善提案を策定・実装した。

**Harness Engineering の核心思想**: AIエージェントの品質を決定するのは「プロンプトの巧さ」ではなく「仕組みの設計」である。

## 提案A: Hooks システムの導入

### 原則

> 「リンターを実行せよ」（プロンプト）と「Hookで例外なく実行」（仕組み）は「ほぼ毎回」と「100%」の差

### ギャップ

- 品質ゲートはスキル呼び出しによる「指示」で実行されていた
- Claude Code の Hooks システム（PreToolUse, PostToolUse, Stop）が未設定

### 実装内容

`.claude/settings.json` に PostToolUse Hook を追加:

```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "bash -c \"[ -f package.json ] && npm run lint --silent 2>&1 | head -30 || exit 0\""
      }]
    }]
  }
}
```

**動作**: `Edit` または `Write` ツール実行後、`package.json` が存在する場合に自動でリントを実行する。プロジェクトのテックスタックに応じてコマンドをカスタマイズすること（例: Python なら `ruff check .`）。

## 提案B: ADR ディレクトリの追加

### 原則

> ADRはタイムスタンプとステータス（Accepted/Superseded/Deprecated）で有効性を構造的に判断可能。内容を書き換えず「置換(supersede)」することで不変性を保持

### ギャップ

- `docs/` に architecture.md はあるが、個々の意思決定記録がなかった
- 「なぜその技術を選んだか」の根拠をエージェントが参照できなかった

### 実装内容

```
docs/adr/
├── README.md        # 運用ルール・命名規則・ステータス定義
└── _template.md     # ADR作成テンプレート
```

ADRテンプレートの構造:
- タイトル / ステータス / 日付
- コンテキスト（なぜ決定が必要だったか）
- 検討した選択肢（メリット・デメリット比較）
- 決定内容と理由
- 結果・トレードオフ

**重要なルール**: 既存のADRは書き換えず、更新が必要な場合は新しいADRを作成して古いADRに `Superseded by [NNNN]` を記載する。

## 提案C: ドキュメント腐敗防止戦略

### 原則

> 記述的な説明文書は必然的にコードに追いつけず腐敗し、エージェントが腐敗情報を真実と誤認する

### ギャップ

- `docs/` の説明的ドキュメントが「実行可能アーティファクト」ではなく「記述テキスト」
- コードと乖離した場合の検出機構がなかった

### 実装内容

1. **フロントマター規則** (`.claude/docs/guidelines/development-guidelines.md` に追加)

   ```yaml
   ---
   last_updated: YYYY-MM-DD
   git_commit: abc1234
   status: current | outdated | draft
   ---
   ```

2. **`/review-docs` コマンドの拡張**

   レビュー観点に以下を追加:
   - 鮮度チェック: `last_updated` が3ヶ月以上前か確認
   - 腐敗リスク: コードに存在するがドキュメントに記載のない機能を検出

3. **「実行可能アーティファクト」優先の原則** (ガイドラインに追加)

   | 代替手段 | 対象情報 |
   |---------|---------|
   | 型定義・スキーマ | データ構造・APIインターフェース |
   | テストコード | 期待される動作・ユースケース |
   | OpenAPI仕様 | APIエンドポイント |
   | CI/CDパイプライン | ビルド・デプロイ手順 |

## 提案D: CLAUDE.md のポインタ化

### 原則

> CLAUDE.md は「ルーティング指示とポインタ」のみ。50行以下が理想

### ギャップ

- 開発時の具体的な注意点が CLAUDE.md に直接記述されていた

### 実装内容

CLAUDE.md の「開発時の注意点」セクション（ホストIP、uv使用、環境変数管理、独立設計の4項目）を `.claude/docs/guidelines/development-guidelines.md` に移動し、CLAUDE.md にはポインタのみを残した:

```markdown
## 開発ガイドライン

- コーディング規約・AI生成コードのアンチパターン・ドキュメント管理戦略: `.claude/docs/guidelines/development-guidelines.md`
- アーキテクチャ意思決定の記録: `docs/adr/`
- フィードバック速度の階層とDoD: `.claude/docs/guidelines/definition-of-done.md`
```

CLAUDE.md は31行に収まり、全てのセクションが「パスへのポインタ」として機能する。

## 提案E: フィードバック速度の階層化

### 原則

> フィードバック速度が品質を決定。最速(ms): PostToolUse Hook → 速(秒): プリコミット → 遅(分): CI → 最遅(時): 人間

### ギャップ

- 品質ゲートは `/add-feature` 完了後（実装の最後）に一括実行
- ファイル編集後の即時フィードバックがなかった

### 実装内容

`.claude/docs/guidelines/definition-of-done.md` に「フィードバック速度の階層」セクションを追加:

| 階層 | タイミング | 手段 | 目標速度 |
|------|----------|------|---------|
| L1: 最速 | ファイル編集直後 | PostToolUse Hook | ミリ秒〜秒 |
| L2: 速 | コミット直前 | プリコミットフック（Lefthook / Husky） | 秒〜数十秒 |
| L3: 遅 | PR作成後 | CI（GitHub Actions 等） | 分 |
| L4: 最遅 | リリース前後 | 人間によるコードレビュー・QA | 時間〜日 |

L1は提案Aの Hooks で実現。L2にはプリコミットフックの設定例（Lefthook）を記載。

## 提案F: セッション起動ルーチンの強化

### 原則

> `git log --oneline` が「何が起きたか」の最高信頼度記録。セッション起動時の標準ルーチンが必要

### ギャップ

- `/resume-work` コマンドに Git ログ参照が含まれていなかった
- フェーズ完了時にコミットを促すガイドがなかった

### 実装内容

1. **`/resume-work` にステップ0を追加**

   作業再開前に `git log --oneline -20` と `git status` で状況を把握するステップ。前回セッションのコミット内容と未コミット変更の有無をユーザーに報告する。

2. **steering スキルにコミット推奨を追加**

   フェーズ完了時に「この時点でコミットしておくと、セッション中断時でも `/resume-work` で正確に再開できます」と提案するステップ。

## 提案G: AI生成コード固有アンチパターン対策

### 原則

> AI生成コードのアンチパターン（any乱用、コード重複、ゴーストファイル、コメント洪水、セキュリティ脆弱性）をカスタムリンターで防止

### ギャップ

- AI固有のアンチパターンに対する具体的な防止策がなかった

### 実装内容

`.claude/docs/guidelines/development-guidelines.md` に5つのアンチパターンとその対策を追加:

| アンチパターン | 症状 | 対策 |
|--------------|------|------|
| ゴーストファイル | 既存修正せず新規ファイル作成 | Glob検索で類似ファイル確認後に作成 |
| TypeScript `any` 乱用 | 型安全性の喪失 | `noImplicitAny: true` + 型ガード |
| コード重複 | 同一ロジック散在 | 実装前に既存コード検索 |
| 過剰コメント | 自明なコードに冗長説明 | 「なぜ」のみコメント |
| セキュリティ脆弱性 | SQLi, XSS, ハードコード認証 | ORM/プリペアド + .env管理 |

## 今後の検討事項

1. **プロジェクト固有の Hook コマンド**: テンプレートのHookは `npm run lint` だが、各プロジェクトのテックスタックに合わせたカスタマイズが必要
2. **Stop Hook の導入**: セッション終了時にテスト全件実行するか、プロジェクト規模による判断が必要
3. **カスタムリンタールール**: AI固有アンチパターンをリンターで自動検出するルールの開発
4. **CI連携**: GitHub Actions のワークフローテンプレートの整備
