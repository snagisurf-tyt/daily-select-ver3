# 技術仕様書 (Architecture Design Document)

## 採用構成: 開発構成（Development Mode）

本アプリケーションは開発構成に基づくアーキテクチャを採用します。理由はPRDで定義された制約条件（単一HTMLファイルでの動作要件、フレームワーク未使用の制約）に完全に合致するためです。本構成により、ユーザーは特別な環境構築やツールチェーンの準備なく、すぐにアプリケーションを利用できます。

## テクノロジースタック

### 言語・ランタイム

| 技術 | バージョン | 選定理由 |
|------|-----------|----------|
| HTML5 | 最新対応ブラウザ | 単一ファイルでのアプリケーション実行要件を満たすため |
| CSS3 | 最新対応ブラウザ | モダンなUI表現とレスポンシブデザイン実現のため |
| JavaScript (ES6+) | 最新対応ブラウザ | フレームワーク未使用の制約とブラウザネイティブAPI利用のため |
| Web API | ブラウザ標準 | ローカルストレージやクリップボード操作などの機能提供のため |

### フレームワーク・ライブラリ

| 技術 | バージョン | 用途 | 選定理由 |
|------|-----------|------|----------|
| Vanilla JS | ブラウザ標準 | アプリケーションロジック実装 | フレームワーク未使用の制約を満たすため |
| LocalStorage API | ブラウザ標準 | データ永続化 | 外部サーバー接続不要でデータ保存可能なため |
| Clipboard API | ブラウザ標準 | 共有リンクのコピー機能 | ユーザー操作の簡便性向上のため |
| Web Components | ブラウザ標準 | UIコンポーネント化（オプション） | 再利用可能なUI部品の実現（必要に応じて使用） |

### 開発ツール

| 技術 | バージョン | 用途 | 選定理由 |
|------|-----------|------|----------|
| 統合開発環境 | 任意 | コーディング | 個人の開発環境に依存しないため任意 |
| ブラウザ開発者ツール | ブラウザ標準 | デバッグ | 単体HTMLファイルでの開発・デバッグに最適 |
| Lighthouse | Chrome標準 | パフォーマンス・アクセシビリティ評価 | 品質保証のため |

## アーキテクチャパターン

### レイヤードアーキテクチャ（シングルページアプリケーション構成）

```
┌─────────────────────────┐
│   UIプレゼンテーション    │ ← ユーザー入力の受付と表示（DOM操作）
├─────────────────────────┤
│   アプリケーションロジック │ ← ビジネスロジックとUI状態管理
├─────────────────────────┤
│   データマネジメント      │ ← ローカルストレージとのデータやり取り
└─────────────────────────┘
```

#### UIプレゼンテーション層
- **責務**: DOM要素の操作、イベントハンドリング、UI状態の反映
- **許可される操作**: アプリケーションロジック層の呼び出し
- **禁止される操作**: データ永続化層への直接アクセス

#### アプリケーションロジック層
- **責務**: スケジュール管理ロジック、参加者情報管理、UI状態の管理
- **許可される操作**: データマネジメント層の呼び出し、UIプレゼンテーション層の更新命令
- **禁止される操作**: DOMの直接操作（UIプレゼンテーション層の関数を利用）

#### データマネジメント層
- **責務**: ローカルストレージとのデータ永続化、データ形式のシリアライズ/デシリアライズ
- **許可される操作**: ブラウザのLocalStorage APIへのアクセス
- **禁止される操作**: アプリケーションロジックの実装、UI状態管理

## データ永続化戦略

### ストレージ方式

| データ種別 | ストレージ | フォーマット | 理由 |
|-----------|----------|-------------|------|
| スケジュールデータ | LocalStorage | JSON文字列 | ブラウザネイティブで利用可能、外部接続不要で永続化可能 |
| 設定情報 | LocalStorage | JSON文字列 | ユーザーの設定情報を保持、外部接続不要で永続化可能 |

### バックアップ戦略

- **頻度**: 自動保存（データ変更時に毎回）
- **保存先**: ブラウザのLocalStorage領域
- **世代管理**: 現在のデータのみ保持（オプションとしてエクスポート機能を提供）
- **復元方法**: localStorage.setItem()でキー指定して上書きすることで復元

## パフォーマンス要件

### レスポンスタイム

| 操作 | 目標時間 | 測定環境 |
|------|---------|---------|
| 日程候補登録 | 50ms以内 | Core i5相当、メモリ8GB、Chrome最新版 |
| UI再描画 | 100ms以内 | 同上 |
| 空き時間登録 | 30ms以内 | 同上 |
| データ保存 | 50ms以内 | 同上 |

### リソース使用量

| リソース | 上限 | 理由 |
|---------|------|------|
| メモリ | 10MB | シングルページアプリケーションとしての軽量性維持 |
| CPU | 5%（アイドル時） | ユーザー操作中の過度な負荷を避ける |
| LocalStorage容量 | 5MB | ブラウザの一般的な制限範囲内で利用 |

## セキュリティアーキテクチャ

### データ保護

- **暗号化**: 保存データは平文（オプションとして軽量な暗号化を検討）
- **アクセス制御**: 同一オリジンポリシーによる自然な保護
- **機密情報管理**: 本アプリケーションでは機密情報を取り扱わない

### 入力検証

- **バリデーション**: 
  - 日付形式チェック（YYYY-MM-DD）
  - 時間形式チェック（HH:mm）
  - 文字数制限（タイトル：100文字以内）
- **サニタイゼーション**: 
  - HTMLエスケープ（innerHTMLへの挿入前）
  - テキストノードによる安全な表示
- **エラーハンドリング**: 
  - 不正な入力に対してはユーザーに分かりやすいエラーメッセージを表示
  - コンソールへのエラーログ出力

## スケーラビリティ設計

### データ増加への対応

- **想定データ量**: 1,000スケジュール、各スケジュールにつき100人の参加者データ
- **パフォーマンス劣化対策**: 
  - 不要なDOM要素の削除（仮想スクロール方式の検討）
  - LocalStorageへのアクセス頻度を抑える（バッチ処理）
- **アーカイブ戦略**: 
  - 明示的なアーカイブ機能は実装しない
  - ユーザーが古いスケジュールを手動で削除可能

### 機能拡張性

- **プラグインシステム**: 実装しない（オールインワンの単体ファイル構成のため）
- **設定のカスタマイズ**: 制限付きのテーマ設定などを localStorage で保持（オプション）
- **API拡張性**: Web API経由でデータのインポート/エクスポート機能を検討（CSVフォーマット）

## テスト戦略

### ユニットテスト
- **フレームワーク**: Jest/Vitest（開発時のみ利用）
- **対象**: 
  - 日付処理関数（DateManagerの機能）
  - ローカルストレージ操作（DataManagerの機能）
  - データバリデーション機能
- **カバレッジ目標**: 80%

### 統合テスト
- **方法**: Playwright/Puppeteerによるブラウザ自動テスト
- **対象**:
  - UI操作とデータ保存の連携
  - 複数人の空き時間��力と集計表示
  - データの永続化と再読み込み

### E2Eテスト
- **ツール**: Playwright
- **シナリオ**:
  - 新規スケジュール作成から共有までの流れ
  - 参加者による空き時間登録と確認
  - ページ再読み込み後のデータ保持確認

## 技術的制約

### 環境要件
- **OS**: 任意（Webブラウザが動作する環境）
- **最小メモリ**: 不要（ブラウザネイティブ機能利用）
- **必要ディスク容量**: 数KB（HTMLファイル単体）
- **必要な外部依存**: なし（ネットワーク接続も不要）

### パフォーマンス制約
- メインスレッドでの重い処理を避ける（必要に応じてsetTimeout分割）
- 1000件以上のデータを一度に描画する場合は仮想スクロールを検討
- 複雑なDOM操作はMutationObserverやバッチ処理により最適化

### セキュリティ制約
- eval()等の動的コード実行関数は使用しない
- 外部から取得したデータをinnerHTMLで直接挿入しない（textContentまたは適切なエスケープを使用）
- localStorage.setItem/getItem使用時に異常な文字列が含まれていないか検査
