# generate-readme 変更履歴セクション追加 仕様書

> 作成日: 2026-03-27
> ステータス: Draft
> 用途: /add-feature への入力

---

## 1. 背景・目的

`/generate-readme` で README.md を生成した後、プロジェクトの変更履歴が README に反映されない。
開発が進むたびに手動で更新するのは非現実的であり、`/generate-readme` 実行時に git log から自動的に変更履歴を生成・更新する仕組みが必要。

## 2. スコープ

### 今回含めるもの
- `/generate-readme` コマンドに「変更履歴の収集・生成」ステップを追加
- README テンプレートに「変更履歴」セクションを追加（ライセンスの直前）
- 前回生成日時を README 末尾の HTML コメントで管理
- 最新 5 件を表示、それ以前は `<details>` タグで折りたたみ
- git log のコミットメッセージを機能単位で要約（AI による要約）

### 今回含めないもの（将来対応）
- `/add-feature` 完了時の自動 README 更新
- 別ファイル（CHANGELOG.md）での管理
- Conventional Commits の厳密なパース

## 3. ユーザーストーリー

- 開発者として、`/generate-readme` を実行するだけで最新の変更履歴が README に反映されてほしい。
- README の読者として、プロジェクトの最近の変更点を一目で把握したい。

## 4. 機能要件

### 4.1 タイムスタンプ管理
- [ ] README.md 末尾に `<!-- readme-generated: YYYY-MM-DDTHH:MM:SS -->` 形式の HTML コメントを埋め込む
- [ ] `/generate-readme` 実行時にこのコメントを読み取り、前回生成日時を取得する
- [ ] コメントが存在しない場合（初回実行）は、変更履歴セクションを「初期リリース」のみで生成する

### 4.2 git log 差分の取得
- [ ] 前回生成日時以降のコミットログを `git log --since="<日時>" --oneline --no-merges` で取得する
- [ ] git リポジトリでない場合はこのステップをスキップし、変更履歴セクションを空で生成する

### 4.3 変更履歴の要約（AI 要約）
- [ ] 取得したコミットログを機能単位でグルーピング・要約する
  - `feat:` プレフィックスのコミットは機能追加として分類
  - `fix:` プレフィックスのコミットはバグ修正として分類
  - 同一機能に関する複数コミットは 1 行にまとめる
- [ ] `.steering/` 配下の `requirements.md` が存在する場合、機能名の参考情報として活用する
- [ ] 各エントリは「日付 + 変更内容の 1 行要約」形式にする

### 4.4 README テンプレートへの反映
- [ ] 「変更履歴」セクションを「ライセンス」セクションの直前に配置する
- [ ] 最新 5 件はテーブル形式で表示する:
  ```markdown
  ## 変更履歴

  | 日付 | 変更内容 |
  |------|---------|
  | 2026-03-27 | ユーザー認証機能を追加 |
  | 2026-03-25 | ダッシュボード画面のレイアウトを改善 |
  | ... | ... |
  ```
- [ ] 6 件目以降は `<details>` タグで折りたたむ:
  ```markdown
  <details>
  <summary>過去の変更履歴</summary>

  | 日付 | 変更内容 |
  |------|---------|
  | 2026-03-20 | 初期リリース |

  </details>
  ```

### 4.5 既存の変更履歴との統合
- [ ] README に既に変更履歴セクションが存在する場合、既存エントリを保持しつつ新しいエントリを先頭に追加する
- [ ] 重複エントリ（同一日付・同一内容）は除去する

## 5. 非機能要件

- パフォーマンス: git log の取得は数秒以内に完了すること
- 堅牢性: git リポジトリでない環境でもエラーにならないこと

## 6. 技術的考慮事項（参考）

> このセクションは壁打ち時点のメモです。詳細設計は `/add-feature` 実行時に `steering/design.md` として生成されます。

### 実装アプローチ（案）
- `.claude/commands/generate-readme.md` に以下を追加:
  - **ステップ 1.5（git log 収集）**: 既存のステップ1（情報収集）の後に挿入
  - **テンプレート変更（ステップ4）**: 変更履歴セクションをテンプレートに追加
  - **タイムスタンプ埋め込み（ステップ4）**: 生成後に末尾コメントを付与

### 既存コードへの影響（想定）
- 変更: `.claude/commands/generate-readme.md` — ステップ追加 + テンプレート拡張

### 注意点・リスク
- コミットメッセージが `feat:` / `fix:` 形式でない場合、要約の質が下がる可能性がある → フォールバックとしてコミットメッセージをそのまま使用する
- git log が大量にある場合（長期間未更新） → 直近 50 件に制限する

## 7. 受け入れ条件

- [ ] `/generate-readme` 実行後の README.md に「変更履歴」セクションが含まれること
- [ ] 変更履歴が機能単位でグルーピング・要約されていること
- [ ] 最新 5 件が表示され、それ以前が折りたたまれていること
- [ ] README 末尾に `<!-- readme-generated: ... -->` コメントが存在すること
- [ ] 2 回目以降の実行で、前回以降の差分のみが追加されること
- [ ] git リポジトリでない環境でもエラーにならないこと
