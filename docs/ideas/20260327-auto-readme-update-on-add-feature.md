# add-feature/add-feature-ui 完了時の README 変更履歴自動更新 仕様書

> 作成日: 2026-03-27
> ステータス: Draft
> 用途: /add-feature への入力

---

## 1. 背景・目的

`/generate-readme` コマンドで README.md に変更履歴セクションを自動生成する機能は実装済みだが、`/add-feature` や `/add-feature-ui` で機能を実装しても README が更新されない。開発者が毎回 `/generate-readme` を手動実行するのは忘れがちで現実的でない。

実装コマンド完了時に、変更履歴セクションだけを自動で差分更新することで、README を常に最新の状態に保つ。

前回の仕様書（`docs/ideas/20260327-readme-changelog-section.md`）で「将来対応」としていた `/add-feature` 完了時の自動 README 更新を今回実装する。

## 2. スコープ

### 今回含めるもの
- `/add-feature` コマンドのステップ8（振り返り）とステップ9（Git操作）の間に「README 変更履歴更新」ステップを追加
- `/add-feature-ui` コマンドにも同じステップを追加
- steering の `requirements.md` から機能名・要約を取得してエントリを生成
- `feat:` と `fix:` に相当する変更のみ記録（メンテナンス系は除外）
- README.md が存在しない場合はスキップ（`/generate-readme` での初回生成を前提）
- コミット前に README を更新し、同一コミットに含める

### 今回含めないもの（将来対応）
- README 全体の再生成（スクリーンショット、バッジ等の更新）
- CHANGELOG.md への分離
- `refactor:`, `docs:`, `chore:` 等のメンテナンス系コミットの記録

### generate-readme との整合性ルール
- `add-feature` が追加したエントリは「正」として扱う
- `/generate-readme` 実行時は、既存の変更履歴エントリを尊重し、前回 `readme-generated` タイムスタンプ以降の新規コミットのみ追加する
- 既存エントリと同一日付のコミットは重複とみなしスキップする
- これにより `add-feature` → 後日 `/generate-readme` の順で実行しても重複が発生しない

## 3. ユーザーストーリー

- 開発者として、`/add-feature` を実行するだけで README の変更履歴が自動更新されてほしい。手動で `/generate-readme` を実行する手間を省きたい。
- README の読者として、プロジェクトの最新機能追加・バグ修正を変更履歴から確認したい。

## 4. 機能要件

### 4.1 ステップの挿入位置

- [ ] `add-feature.md`: ステップ8（振り返りとドキュメント更新）の後、ステップ9（Git操作）の前に「ステップ8.5: README 変更履歴の自動更新」を挿入
- [ ] `add-feature-ui.md`: 同上

### 4.2 README 存在チェック

- [ ] README.md がプロジェクトルートに存在するか確認する
- [ ] 存在しない場合: このステップをスキップし、次のステップ（Git操作）に進む
- [ ] 存在する場合: 変更履歴セクションの有無を確認する

### 4.3 変更履歴セクションの有無チェック

- [ ] README.md 内に `## 変更履歴` セクションが存在するか確認する
- [ ] 存在しない場合: このステップをスキップ（`/generate-readme` で初回生成されていないため）
- [ ] 存在する場合: エントリ追加処理に進む

### 4.4 エントリ内容の生成

- [ ] ステアリングディレクトリの `requirements.md` を読み、以下を抽出する:
  - **機能名**: 仕様書のタイトルまたは機能名フィールド
  - **変更種別**: 新規機能追加 → `feat`、バグ修正 → `fix`（requirements.md の内容から判定）
- [ ] `feat:` または `fix:` に該当する変更のみエントリを生成する
  - `refactor:`, `docs:`, `chore:` 相当の作業（例: リファクタリングのみ、ドキュメント修正のみ）の場合はエントリを追加しない
- [ ] エントリ形式: `| YYYY-MM-DD | 機能名・変更内容の1行要約 |`
  - 日付は現在日付を使用

### 4.5 README.md への差分更新

- [ ] `## 変更履歴` セクション内のテーブルの先頭行（ヘッダー行の直後）に新しいエントリを挿入する
- [ ] 最新5件の管理:
  - テーブル本体（`<details>` の外）のエントリが5件を超える場合、最も古いエントリを `<details>` 内に移動する
  - `<details>` セクションが存在しない場合は新規作成する
- [ ] `<!-- readme-generated: ... -->` タイムスタンプは更新しない（generate-readme 専用のタイムスタンプのため）
- [ ] Edit ツールで差分更新を行う（Write による全体上書きは行わない）

### 4.6 コミットへの含め方

- [ ] ステップ9（Git操作）の 9-B でステージングする際、README.md の変更も自動的にステージングに含める
- [ ] 追加のコミットは作成しない（実装コードと同一コミットに含める）

## 5. 非機能要件

- パフォーマンス: requirements.md の読み込みと Edit による差分更新のみなので、追加処理時間は無視できるレベル
- 堅牢性: README.md が存在しない、変更履歴セクションがない等のケースでエラーにならずスキップすること
- 冪等性: 同じ機能で複数回実行しても重複エントリが追加されないこと（同一日付・同一内容のチェック）

## 6. 技術的考慮事項（参考）

> このセクションは壁打ち時点のメモです。詳細設計は `/add-feature` 実行時に `steering/design.md` として生成されます。

### 実装アプローチ（案）

両コマンドファイルに以下のステップを追加する:

```markdown
## ステップ8.5: README 変更履歴の自動更新

1. README.md の存在を確認する。存在しない場合はこのステップをスキップ。
2. README.md 内に `## 変更履歴` セクションが存在するか確認する。存在しない場合はスキップ。
3. ステアリングディレクトリの `requirements.md` を読み、機能名と変更種別を特定する。
4. 変更種別が `feat` または `fix` に該当する場合のみ、以下を実行:
   a. 新しいエントリ `| YYYY-MM-DD | [変更内容の1行要約] |` を生成する。
   b. README.md の変更履歴テーブルのヘッダー直後に Edit ツールで挿入する。
   c. テーブル本体が5件を超える場合、最古のエントリを <details> 内に移動する。
   d. 重複チェック: 同一日付・同一内容のエントリが既に存在する場合は追加しない。
5. 変更種別が上記以外（refactor, docs, chore 等）の場合はスキップ。
```

### 既存コードへの影響（想定）
- 変更: `.claude/commands/add-feature.md` — ステップ8.5 を追加
- 変更: `.claude/commands/add-feature-ui.md` — ステップ8.5 を追加
- 変更: `.claude/commands/generate-readme.md` — ステップ1.5-D の重複判定ロジックを強化（既存エントリと同一日付のコミットをスキップ）

### 注意点・リスク
- requirements.md から変更種別（feat/fix）を正確に判定できない場合がある → デフォルトは `feat`（新機能追加）として扱う
- README.md の変更履歴テーブルのフォーマットが手動編集で崩れている場合、Edit が失敗する可能性がある → セクションの存在チェックで最低限のバリデーションを行う
- `add-feature` が品質ゲート失敗でエスカレーションした場合、ステップ8.5 には到達しないため README は更新されない（意図通り）

## 7. 受け入れ条件

- [ ] `/add-feature` 完了後、README.md の変更履歴セクションに今回の実装内容が追加されていること
- [ ] `/add-feature-ui` 完了後も同様に変更履歴が追加されていること
- [ ] 変更履歴エントリが実装コードと同一の git コミットに含まれていること
- [ ] README.md が存在しない場合、エラーにならずスキップされること
- [ ] 変更履歴セクションが存在しない場合、エラーにならずスキップされること
- [ ] `refactor:` 相当の作業では変更履歴が追加されないこと
- [ ] 同一内容の重複エントリが追加されないこと
- [ ] 最新5件がテーブル表示、6件目以降が折りたたみに正しく配置されること
- [ ] `/add-feature` 後に `/generate-readme` を実行しても、同一機能のエントリが重複しないこと
- [ ] `generate-readme.md` の変更履歴統合ロジック（ステップ1.5-E）に、既存エントリの同一日付重複スキップルールが追加されていること
